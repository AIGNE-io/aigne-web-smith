# 素材の準備

優れたウェブサイトの基盤は、`aigne web generate` を実行する前に築かれます。このガイドでは、WebSmith が要件を反映し、適切な証拠を引用し、修正サイクルを減らしたページを構築できるよう、企画書、ソースドキュメント、証明素材を収集するための具体的なチェックリストを提供します。

## パイプラインの仕組み

```d2
direction: right

Gather: {
  label: "企画書とアセットを収集"
  shape: rectangle
}

Organize: {
  label: "sources/ を整理"
  shape: rectangle
}

Configure: {
  label: "config.yaml を更新"
  shape: rectangle
}

Generate: {
  label: "`aigne web generate` を実行"
  shape: rectangle
}

Review: {
  label: "ページをレビューして反復"
  shape: rectangle
}

Gather -> Organize -> Configure -> Generate -> Review
```

各ステップは次のステップにつながります。雑然とした入力は一般的な出力につながりますが、厳選されたコンテンツキットがあれば、WebSmith は最初のパスから信頼性のある文章を生成できます。

## ステップ 1. 基本事項を把握する

4 つの質問に答える短い企画書 (Markdown または DOCX で可) を作成します。

| 質問 | なぜ重要か | 例 |
| :---------------- | :-------------------------------------------------------------------------------------------------- | :---------------------------------------------------------------------- |
| **オーディエンス** | トーン、語彙、証明資料は、ページの読者に合わせる必要があります。 | 「成長段階のフィンテック企業の創業者とそのエンジニアリングリーダー」 |
| **課題** | あなたが解決する課題を明確にします。 | 「手作業のオンボーディングにより、10時間以上の反復的なコンプライアンスレビューが発生している」 |
| **差別化要因** | コピーが競合他社と同じようにならないようにします。 | 「地域のデータレイク上に構築された KYC + KYB 自動化機能を備えた唯一のプラットフォーム」 |
| **主要な CTA** | すべてのセクションを単一のコンバージョンアクションに向けます。 | 「20分間のインテグレーションレビューを予約する」 |

このファイルをプロジェクト内に保存し、`sourcesPath` に追加できるようにします。

## ステップ 2. 参考コンテンツとアセットを収集する

WebSmith は、あなたが提供したものしか使用できません。以下を `sources` ディレクトリなどにまとめます。

### 推奨されるコンテンツタイプ

| コンテンツタイプ | 説明 | 例 |
| :-------------------- | :--------------------------------------------------------------------------------------------------------------------------------------- | :---------------------------------------------------------------------------- |
| **製品ドキュメント** | 機能の内訳、アーキテクチャノート、API リファレンス、価格説明。 | `docs/product-overview.md`、`api/quickstart.md` |
| **マーケティングプラン** | ポジショニング、メッセージングハウス、キャンペーン企画書、競合他社サマリー。 | `marketing/positioning.pdf`、`brand/voice.md` |
| **事業計画** | ミッション、ビジョン、ロードマップ、資金調達のマイルストーン、リーダーシップの経歴。 | `company/mission.txt`、`investor/roadmap.pptx` |
| **既存のコンテンツ** | ブログ記事、FAQ、変更履歴など、トーンや繰り返し使えるストーリーを示すもの。 | `blog/*.md`、`faq.md`、`release-notes/2024-02.md` |
| **メディアファイル** | ロゴ、製品のスクリーンショット、チーム写真、チャート。 | `assets/logo.svg`、`screenshots/dashboard.png` |

### サポートされているフォーマット

| カテゴリ | フォーマット |
| :------------ | :--------------------------------------------------------------------------------------------------------- |
| **テキスト** | `.md`, `.txt`, `.html`, `.json`, `.yaml`, `.xml` |
| **ドキュメント** | `.pdf`, `.doc`, `.docx`, `.xls`, `.xlsx`, `.ppt`, `.pptx` |
| **画像** | `.jpg`, `.jpeg`, `.png`, `.gif`, `.svg`, `.webp` |
| **コード** | `.js`, `.ts`, `.py`, `.go`, `.rs`, `.java`、その他スニペット抽出用の主要な言語のほとんど |

## ステップ 3. すべてを再利用可能に整理する

あなた (そして WebSmith) がどこに何があるかを把握できるように、ファイルを論理的にグループ化します。

```sh project-sources/ icon=lucide:folder-tree
project-sources/
├── 01_briefs/
│   ├── product-overview.md
│   └── audience-matrix.md
├── 02_proof/
│   ├── testimonials.md
│   └── security-metrics.xlsx
├── 03_assets/
│   ├── logo.svg
│   └── dashboard.png
└── 04_content/
    ├── blog/
    └── faq.md
```

どのディレクトリがオーディエンス、証明資料、またはメディアに対応しているかがわかるように、説明的なフォルダ接頭辞を使用します。

## ステップ 4. `config.yaml` でソースを設定

`sourcesPath` 配列で、先ほど整理したディレクトリ (または特定のファイル) を指定します。これは最も重要な設定パラメータです。

```yaml config.yaml icon=lucide:file-code
sourcesPath:
  - ./project-sources/01_briefs
  - ./project-sources/02_proof
  - ./project-sources/03_assets
pagePurpose:
  - saas
targetAudienceTypes:
  - businessOwners
  - developers
rules: >
  Highlight 40% cost savings backed by customer quotes. Mention SOC 2 + ISO 27001.
```

`aigne web generate` を実行すると、WebSmith はこれらのフォルダを再帰的に読み込み、ファイルをチャンク化し、コピーを書き込みレイアウトを組み立てる際にそれらを引用します。

## タイムラインに合った準備モードを選択する

### スターターキット (最小限の実行可能な入力)

- 1つのよく書かれた README (約500語以上) といくつかのスクリーンショット。
- そのドキュメントと画像を `sourcesPath` に追加します。
- `aigne web generate` を実行し、出力をレビューして一度反復します。
- デモサイトを迅速に必要とするが、後で磨きをかけることを想定している場合に役立ちます。

### 戦略的企画書 (推奨されるデフォルト)

1. **オーディエンス**、**課題**、**差別化**、**CTA** を文書化します。
2. 各ペルソナに対して、機能的および感情的な証明資料を含むバリューマトリックスを構築します。
3. WebSmith にコピーを書かせる前に、各ページセクションを意図ごとにアウトライン化します。
4. お客様の声、メトリクス、製品のスクリーンショットを別々のファイルとしてパッケージ化します。
5. 各生成パスの後、すべてのセクションがその意図を果たしていることを確認し、ソースを修正して再実行します。

### モジュラーナレッジシステム (上級チーム向け)

- 専門知識を焦点を絞った Markdown ファイル (基盤、製品、証明資料、ペルソナ) に分割します。
- 構造例:

```sh
knowledge-base/
├── foundation/
│   ├── mission.md
│   └── brand-voice.md
├── products/
│   ├── payments-overview.md
│   └── payments-technical-specs.md
├── proof-points/
│   ├── case-study-fintech-x.md
│   └── g2-reviews.md
└── audiences/
    ├── developer-persona.md
    └── operator-persona.md
```

- プロジェクトごとにディレクトリを組み合わせて、すべてのサイトが単一の情報源からデータを取得するようにします。
- 1つのモジュール (例: `case-study-fintech-x.md`) を更新すると、それを参照する将来のすべての生成が自動的に改善されます。

## 品質チェックリスト

- [ ] すべての主張には、ソースファイルにデータポイント、引用、またはメトリクスがある。
- [ ] ペルソナと CTA は、マーケティングの誇張表現とは別に文書化されている。
- [ ] アセットには説明的な名前が付けられている (`dashboard-dark.png` であり、`image1.png` ではない)。
- [ ] 機密情報や内部専用ファイルは除外されている (WebSmith はあなたが指定したすべてを読み取る)。
- [ ] ソースドキュメントは知識に焦点を当て、スタイルガイダンスは `rules` に記述する。

## 最良の結果を得るためのヒント

WebSmith は、優れたウェブサイトを一度で生成するように設計されていますが、品質は提供する素材に依存します。

- **具体的な詳細を提供する。** ソース素材が具体的であるほど（データポイント、引用、指標）、ウェブサイトはより説得力のあるものになります。
- **コンテンツを焦点を絞ったファイルに分割する。** 一つの巨大なドキュメントではなく、トピックを小さなファイルに整理します。これにより、WebSmith はソースを正確に引用し、より構造化されたコンテンツを生成できます。
- **コンテンツと指示を分離する。** 知識はソースファイルに入れますが、レイアウトとトーンの指示は `rules` フィールドに入れます。
